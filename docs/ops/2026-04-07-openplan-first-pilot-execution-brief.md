# OpenPlan First Pilot Execution Brief — 2026-04-07

**Owner:** Bartholomew Hale (COO)  
**Status:** active execution brief  
**Purpose:** turn the approved supervised-pilot posture into one concrete pilot motion the team can run with minimal ambiguity.

## Executive summary

OpenPlan's first pilot should be executed as:

**one supervised rural-agency planning workflow centered on project continuity and report traceability**

The target is not to prove the whole OpenPlan thesis.
The target is to prove one truthful sentence:

> OpenPlan helped a real planning team complete a real workflow with more continuity, more traceability, and less confusion.

## Recommended first pilot shape

### Agency profile
Best-fit first pilot user:
- rural county,
- RTPA / transportation commission,
- or small public-agency planning team,
- with one primary project lead and one secondary reviewer.

### Workflow profile
Best-fit first pilot workflow:
- one Active Transportation Plan update,
- one corridor safety or multimodal planning workflow,
- or one scoped programming / plan-package workflow

with the following emphasis:
1. project setup and workspace targeting,
2. plan / model / program continuity,
3. report packet assembly and source traceability,
4. one bounded scenario or comparison review,
5. human-supported final handoff.

## Exact first-pilot promise

The first pilot should promise only this:
- structured project continuity,
- auditable report traceability,
- guided supervised onboarding,
- clear operator support,
- bounded scenario coordination where relevant.

It should **not** promise:
- validated forecasting claims,
- automated compliance/legal judgment,
- broad self-serve onboarding,
- enterprise replacement,
- unlimited modeling services.

## First session operator script

### Step 1: confirm scope
Before setup, confirm:
- exact project name,
- exact workflow being piloted,
- exact success boundary,
- named primary operator,
- named secondary reviewer.

### Step 2: open the right workspace
The operator must confirm:
- correct workspace selected,
- correct project context,
- correct role/access posture,
- no ambiguity about what records belong to the pilot.

### Step 3: establish the workflow spine
During the pilot, OpenPlan should clearly hold:
- project record,
- plan record,
- model record if used,
- program/report linkages if used,
- source references and supporting context.

### Step 4: produce one report-centered outcome
The pilot should end with one tangible artifact such as:
- report packet,
- draft planning memo,
- traceable scenario comparison,
- or documented project-status/report handoff.

### Step 5: close with operator review
Before calling the pilot successful, ask:
- did the workflow remain understandable?
- did the team stay in the right project context?
- were the report links and sources traceable?
- did support feel clear and human?
- would the agency trust this for the next similar workflow?

## Proof-of-value checkpoints

The pilot is only counted as successful if all of these are true:

### Checkpoint 1: context integrity
- correct workspace was used,
- no operator confusion about what was being billed or edited,
- no project-context drift.

### Checkpoint 2: continuity
- project / plan / model / program relationships stayed coherent,
- the pilot user could navigate the workflow without losing the thread.

### Checkpoint 3: traceability
- report output or reporting context clearly linked back to source records,
- reviewer could inspect where key output context came from.

### Checkpoint 4: supportability
- onboarding was understandable,
- support requests had a clear resolution path,
- pilot did not depend on hidden heroic behavior.

### Checkpoint 5: extension viability
- pilot user can name the next adjacent workflow they would trust OpenPlan to support.

## Failure conditions

Treat the pilot as failed or incomplete if any of the following happen:
- workspace/project targeting feels ambiguous,
- report output is not traceable enough for a planner to trust,
- the team needs to overstate capabilities to keep the pilot appealing,
- support becomes improvised or inconsistent,
- the workflow depends on product behavior that is not actually stable.

## Success artifact package

At pilot close, the team should produce:
1. pilot summary note,
2. workflow completed / not completed,
3. traceability proof artifact or screenshots,
4. operator friction list,
5. recommended next pilot extension or stop decision.
