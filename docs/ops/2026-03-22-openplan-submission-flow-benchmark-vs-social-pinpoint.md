# OpenPlan Engagement — Submission Flow Clarity Benchmark vs. Social Pinpoint / Open Point

**Date:** 2026-03-22  
**Owner:** Camila Reyes  
**Lane:** OpenPlan Priority #2 — Engagement support only  
**Posture:** bounded UX/design support artifact only; no code edits, no dashboard work, no broad redesign

---

## Purpose

This memo benchmarks the **current / current-planned OpenPlan engagement submission flow** against the submission-flow expectations implied by **Social Pinpoint / Open Point**.

The goal is not to pretend OpenPlan already has more feature breadth.

The goal is to make the **submission flow we do have** feel:
- more trustworthy,
- more legible,
- more planning-grade,
- more beautiful,
- and more reassuring to complete

than the generic civic SaaS baseline.

---

## Honest benchmark posture

### What Social Pinpoint / Open Point appears to do well
From the reviewed product surface, Social Pinpoint-style engagement tools appear strong at:

1. **Immediate orientation**  
   The user is usually placed inside a clearly branded civic project context before being asked to contribute.

2. **Tool-specific framing**  
   Each participation mode tends to signal what kind of input is expected (poll, map comment, Q&A, idea board, etc.).

3. **Polished trust cues**  
   The interfaces read as institutionally credible, with calm layouts, municipal language, and obvious project identity.

4. **Visually finite flows**  
   Even when the platform is modular, each entry surface tends to feel bounded: the user understands where they are and what they are doing.

5. **Generic but competent reassurance**  
   The products signal legitimacy and reduce uncertainty, even when the design is somewhat generic SaaS polish.

---

## Where OpenPlan is currently thinner

OpenPlan’s current engagement lane already has meaningful substance:
- public engagement portal / share page,
- public read/submit APIs,
- operator share controls,
- bulk moderation UI,
- category/item handling,
- export route,
- handoff/report bridge posture.

But on **submission-flow clarity specifically**, the thinner areas are likely:

1. **less pre-submit orientation language**  
   The user may understand that they can submit, but not yet feel clearly guided about what a useful submission looks like.

2. **weaker distinction between required contribution and optional metadata**  
   If optional context fields visually compete with the main submission, the flow feels heavier than it actually is.

3. **less explicit after-submit reassurance**  
   If the user is not told how moderation/review works, trust drops.

4. **less emotionally calm framing**  
   A planning-grade submission flow should feel composed and civic-serious, not merely functional.

5. **less “why this matters” language**  
   Social Pinpoint-style tools are not better because they are magical; they are better because they feel institutionally prepared to receive public input.

---

## The key strategic insight

OpenPlan does **not** need feature parity to outperform Social Pinpoint on this specific flow.

It can feel better by being:
- more specific,
- more human,
- more transparent,
- and more planning-serious

at the exact moment the user decides whether to submit.

In other words:

> We do not need more modules to make the current submission experience feel better.  
> We need better hierarchy, better framing, better optionality signals, and better trust language.

---

## What OpenPlan should do better on this flow

### 1. Start with project/campaign meaning, not raw fields

#### Social Pinpoint-style expectation
The user lands in a civic project context that feels legitimate before interacting.

#### OpenPlan opportunity
OpenPlan should make the submission flow feel more planning-grade by opening with:
- campaign title,
- project linkage,
- a one-sentence summary,
- and one short line on what kind of input is most useful.

#### Why this can beat Social Pinpoint
Because OpenPlan can sound **more specific and less generic**.

Not:
- “Provide your feedback.”

Instead:
- “Share a specific observation, issue, or idea related to this project area.”

That shift makes the platform feel more serious and more grounded in planning work.

---

### 2. Make the main ask feel simpler than the page looks

#### Social Pinpoint-style expectation
Users expect a structured civic participation tool but not an exhausting form.

#### OpenPlan opportunity
The form should visually communicate:
- **one main thing is required**,
- everything else helps but is optional unless stated otherwise.

#### Recommendation
The primary content field should dominate visually.
All secondary fields should appear clearly beneath it as supporting context.

This makes the flow feel lighter without removing capability.

---

### 3. Use stronger optional vs. required signaling

#### Social Pinpoint-style expectation
A polished participation tool should not make the user guess what they can skip.

#### OpenPlan opportunity
Required and optional must be visibly signaled in-line.

### Recommended labeling style
- **Your input** *(required)*
- **Category** *(optional)*
- **Location** *(optional)*
- **Name** *(optional unless follow-up is requested)*
- **Email for follow-up** *(optional)*

### Why this matters
Users abandon forms when they assume optional context is mandatory.

OpenPlan can outperform the generic benchmark by being **more explicit and calmer**.

---

### 4. Explain what happens after submit

#### Social Pinpoint-style expectation
Institutional trust comes from knowing the input enters a real process.

#### OpenPlan opportunity
The submission flow should explicitly state:
- submissions may be reviewed,
- submissions may be categorized/moderated,
- submissions may inform project reporting or engagement summaries,
- direct responses are not always guaranteed unless follow-up is requested.

### Why this can beat Social Pinpoint
Because OpenPlan can make the planning/reporting connection more explicit.

This is a unique advantage:
- OpenPlan is not just a civic comment tool,
- it is a planning-operating system with a visible handoff/report bridge.

That should be reflected in the trust language.

---

### 5. Make the tone calmer and more planning-grade

#### Social Pinpoint weakness to avoid
Social Pinpoint-style systems can feel polished but somewhat generic and modularized to the point of sameness.

#### OpenPlan opportunity
The submission flow should feel:
- quieter,
- more exact,
- more civic-serious,
- more region/project grounded.

### Tone guidance
Prefer:
- plain civic language,
- quiet confidence,
- specific planning intent,
- low-drama reassurance.

Avoid:
- startup-style enthusiasm,
- generic “your voice matters” clichés without context,
- overly transactional field language.

---

## Recommended submission hierarchy

### Section A — What this is
At the top:
- campaign title
- linked project name
- one-sentence campaign purpose
- one short line: what kind of input is helpful

### Section B — Your contribution
Primary prompt first:
- **What would you like us to know?**
- or **What issue, opportunity, or concern are you seeing?**

This is the center of gravity.

### Section C — Helpful context
Only after the main submission:
- category *(optional)*
- location *(optional)*
- supporting context *(optional)*

### Section D — Follow-up information
Last, and clearly secondary:
- name *(optional / as needed)*
- email *(optional / as needed)*

### Section E — What happens next
Short reassurance block before submit or in confirmation state.

---

## Recommended helper text

### Top-of-form helper copy
> Share a specific issue, observation, or idea related to this campaign. A short, clear note is enough.

### Main field helper copy
> Focus on what you noticed, where it matters, and why it is important.

### Optional field helper copy
> These details help us sort and review submissions, but they are not required unless noted.

### Pre-submit expectation copy
> Submissions may be reviewed, categorized, and summarized as part of project engagement and reporting.

### Post-submit confirmation copy
> Thank you. Your input has been received and will enter the campaign review process.

## How OpenPlan should feel better than Social Pinpoint on this flow

This is the explicit target.

### OpenPlan should feel better by being:

#### More specific
The flow should feel like it belongs to a real planning project, not a generic participation module.

#### More legible
The user should immediately understand:
- what this is,
- what is required,
- what is optional,
- and what happens next.

#### More planning-grade
The experience should connect the submission to a real civic/planning process, not just a comment inbox.

#### More trustworthy
Trust comes from:
- calm structure,
- explicit expectations,
- disciplined language,
- and visible seriousness.

#### More beautiful in restraint
Not flashier.  
Better by being cleaner, quieter, and more deliberate.

---

## Smallest bounded moves to prioritize first

If only a few clarity moves are taken from this memo, they should be these:

### 1. Add a top-of-form orientation block
Explain campaign purpose, useful input type, and time expectation.

### 2. Visually prioritize the main submission field
Make the actual contribution feel like the primary task.

### 3. Label optional fields explicitly
Do not force the user to infer optionality.

### 4. Add a short “what happens next” statement
Before and/or after submit.

### 5. Use planning-grade civic copy
Replace generic participation language with calm, specific, project-linked language.
