# OpenPlan First Pilot Target — 2026-04-07

**Owner:** Bartholomew Hale (COO)  
**Status:** active execution target  
**Purpose:** convert the approved supervised-pilot posture into one concrete, credible first-pilot target the team can actually execute.

## Executive decision

The first OpenPlan pilot target should be:

**a rural county / RTPA planning team using OpenPlan for one supervised, report-centered planning workflow**

This is the strongest first target because it matches:
- Nat Ford's real credibility,
- OpenPlan's current evidence-backed surface,
- and the kind of workflow where auditable continuity matters more than flashy automation.

## Recommended agency profile

Target profile:
- small rural county,
- RTPA or transportation commission,
- or a local public-agency planning team with limited staff capacity,
- where one planner or project lead can work closely with Nat Ford in a supervised setup.

## Recommended first workflow

Primary workflow:
1. create or organize the project workspace,
2. maintain project / plan / model / program continuity,
3. generate and review a report packet with source traceability,
4. support one bounded scenario or analysis comparison,
5. close the loop with a human-supported handoff.

Research-aligned preferred shape for that bounded comparison:
- one corridor, ATP/RTP, or programming-oriented scenario,
- one clear accessibility or comparable decision-support readout,
- and, where feasible, one small equity or public-review-ready narrative output that remains easy to explain.

This should be framed as:
- one real planning workflow,
- one named project context,
- one supervised operator path,
- and one clear end-to-end success story.

## Best-fit use cases

Best current candidates:
- Active Transportation Plan update support,
- corridor safety / multimodal analysis support,
- scoped RTP / programming coordination support,
- project-package continuity for report assembly and revision tracking.
- scenario-centered planning support where one comparison can be explained with honest assumptions and traceable source context.

## Why this target is the right first move

### 1. It matches current product proof
OpenPlan is strongest today in:
- authenticated planning continuity,
- report traceability,
- scenario coordination,
- and guided workflow structure.

### 2. It matches Nat Ford's actual market trust
Nathaniel already has deep credibility with:
- rural counties,
- transportation commissions,
- and planning workflows where structured documentation matters.

### 3. It avoids the dangerous first-pilot traps
This target does **not** require us to prove:
- universal self-serve onboarding,
- broad forecasting validation,
- full compliance automation,
- or an enterprise replacement thesis.

## Explicit first-pilot boundaries

The first pilot is **in scope** for:
- one agency or one project context,
- one or two named planners/operators,
- one bounded planning workflow,
- supervised onboarding,
- human-guided support,
- report-centric proof of value.
- one bounded scenario/accessibility comparison where the evidence is real and legible.

The first pilot is **out of scope** for:
- broad public launch rhetoric,
- unlimited modeling requests,
- full CEQA / compliance automation claims,
- multi-department rollouts,
- unsupervised onboarding,
- claims that OpenPlan is already a universal planning OS for any agency.

## Success definition

The first pilot is successful if the pilot user can honestly say:
- we stayed in the correct project/workspace context,
- the workflow was understandable,
- the report and supporting links were traceable,
- the support path was clear,
- and this is worth extending into a second scoped use.

## Required operator posture

Nat Ford should run this pilot as:
- supervised,
- hands-on,
- evidence-accurate,
- and intentionally narrow.

The operator rule is simple:
- **do not try to impress with breadth**,
- **prove trust, continuity, and usefulness instead**.

## Immediate execution implications

This target means the next implementation and ops work should favor:
1. report/planning orchestration,
2. clean project/workspace setup,
3. traceable handoff and support,
4. planner-safe language and boundaries,
5. repeatable onboarding steps.

## Recommended working sentence

Use this internally as the first-pilot sentence:

> OpenPlan's first pilot should be a supervised rural-agency planning workflow centered on project continuity and report traceability, not a broad self-serve product proof.

## Bottom line

The right first pilot is not the biggest one.

It is the one most likely to produce a truthful sentence like:

> "This helped a real planning team complete a real workflow with more continuity, more traceability, and less confusion."
