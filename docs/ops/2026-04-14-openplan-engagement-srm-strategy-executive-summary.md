# OpenPlan Engagement + Stakeholder Operations Strategy — Executive Summary

Date: 2026-04-14  
Audience: Bartholomew Hale, Nathaniel Ford Redmond, product/engineering leads  
Status: Planning summary only, not execution approval

## Core recommendation

Do **not** rebuild OpenPlan as a separate OpenPoint clone.

Instead, expand OpenPlan into one unified platform with three connected layers:

1. **Planning OS core**  
   Projects, plans, programs, reports, scenarios, RTP cycles, funding, county workflows, controls.

2. **Engagement layer**  
   Public campaigns, stage-based participation, map/form/poll/Q&A tools, moderation, reporting-back.

3. **Stakeholder operations layer**  
   Stakeholder profiles, interaction history, commitments, issues, tasks, outreach continuity.

## Why this is the right move

OpenPlan already has a real product spine:
- authenticated workspace flows,
- projects/plans/programs/models/reports/scenarios,
- engagement campaigns and public/share intake,
- geospatial surfaces,
- report traceability,
- pilot-bounded operations and positioning.

That means the best path is **additive expansion**, not replacement.

A clone strategy would create rewrite pressure, duplicate architecture, and reduce differentiation.

A unified-platform strategy compounds current repo value and creates a much stronger long-term category position.

## Product thesis

OpenPlan should become:

> a planning operations platform that unifies project workflow, public engagement, stakeholder continuity, and evidence-grade reporting inside one auditable workspace.

## Strategic principles

1. **No big-bang rewrite**  
   Extend the current repo and schemas.

2. **Project stays the anchor object**  
   Engagement, stakeholders, reporting, and controls should all resolve back to projects and linked planning objects.

3. **One evidence spine**  
   Audit, provenance, exports, and report linkage should be shared across modules.

4. **Planning-native, not generic SaaS**  
   Build planning/public-sector workflows, not a generic CRM plus generic PM board.

5. **AI after record integrity**  
   AI should accelerate summaries and next actions only after moderation, evidence, and audit foundations are solid.

## Recommended sequence

### Phase 0: architecture and preservation
- lock canonical domain ownership
- prevent duplicate object models
- define migration/compatibility rules

### Phase 1: deepen engagement
- add campaign stages
- add multiple participation tools
- improve public pages and moderation
- add public reporting-back

### Phase 2: build the project control room
- deadlines, decisions, issues, commitments, meetings
- linked engagement/report/scenario/funding posture
- project as operational hub

### Phase 3: add stakeholder operations
- stakeholder profiles and organizations
- interaction history
- commitments/issues/tasks
- CSV import and continuity across campaigns/projects

### Phase 4: unify into reports and evidence
- comment summaries
- response matrices
- stakeholder logs
- project/control outputs
- traceable exports

### Phase 5: add operations copilot and AI assist
- workspace ops summary
- action queue
- AI summaries, clustering, overdue follow-up surfacing
- all grounded in real records and citations

## What this preserves

This strategy protects existing investments in:
- projects,
- engagement campaigns,
- report handoff,
- scenarios,
- RTP direction,
- workspace/auth model,
- pilot-bounded product truth.

## Biggest risk to avoid

The main risk is building three disconnected systems:
- planning app,
- engagement app,
- mini CRM.

The mitigation is strict shared anchors:
- project,
- report/evidence,
- audit/provenance,
- unified control-room summaries.

## Bottom line

The strongest move is to make OpenPlan the system that connects:

**public input -> moderation -> stakeholder follow-up -> project decisions -> report outputs -> auditable evidence**

That is more valuable than building a standalone clone of either research lane.
