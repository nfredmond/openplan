## Buyer-Safe Boundary

This package uses Nevada County / NCTC geography, public-data-style proofing, and the existing OpenPlan demo materials to make the release-to-sale offer concrete. It is not an endorsement by NCTC, not an active partnership, not a procurement signal, and not a claim that NCTC has reviewed, approved, or supplied non-public data.

## Purpose

Prepare the first buyer-specific release-to-sale package around one concrete rural RTPA use case: a guided OpenPlan demo workspace that shows how a county-scale RTP workflow can keep geography, model evidence, caveats, reports, and buyer review in one workspace.

The goal is a 30-minute walkthrough and a scoped pilot conversation, not a broad self-serve SaaS pitch.

## Buyer Fit

Nevada County / NCTC is a strong first-buyer analogue because the current demo already matches a rural RTPA planning shape:

- county-scale transportation planning with state-route, tourism, evacuation, complete-streets, transit, and access considerations;
- a realistic RTP/report-packet workflow where board materials, evidence, caveats, and export artifacts matter;
- staff-capacity pressure where an evidence-backed workbench can reduce manual assembly work;
- public-data-first proofing using OSM, Census/ACS, LODES, public count data, and source manifests; and
- a natural path from RTP evidence into grants, engagement, and project-priority workflows.

This fit does not mean the current screening run is a validated planning-grade model. It means the demo is useful for deciding whether a supervised pilot is worth scoping.

## First Workflow Recommendation

Lead with the RTP/report packet workflow.

Recommended first proof: show the NCTC 2045 RTP proof-of-capability workspace, the existing-conditions/travel-patterns chapter, the source manifest, the validation summary, the caveats, and the generated report/document posture.

Why this is the right wedge:

- it is the clearest fit with the existing NCTC demo one-pager;

## Scoped Pilot Ask

Ask for permission to scope one supervised pilot, not a full platform rollout.

Suggested pilot shape:

- Primary workflow: RTP/report packet workflow.
- Term: 30-90 days from kickoff.
- Workspace: one managed demo/pilot workspace with one named owner and one decision reviewer.
- Inputs: public sources first, then buyer-approved project lists, plan materials, or local context if provided.
- Outputs: one configured workflow proof, one guided review session, one export or report artifact, one caveat/provenance summary, and one closeout memo.
- Required preconditions: data sensitivity confirmed, partnership language approved, support contact named, and SOW/service-schedule fields filled before paid activation.

Pilot decision question:

> Would it be useful to scope a short supervised RTP/report workflow pilot around one existing plan chapter, corridor, or board-packet deliverable, using public data first and adding locally reviewed inputs only if you want to proceed?

## Follow-Up Checklist

- Confirm the first workflow: RTP/report packet unless the buyer explicitly chooses another workflow.
- Confirm whether the buyer wants a guided screen share or a reviewed signed-in demo login.
- Send buyer one-pager, NCTC demo one-pager, and caveat sheet before or after the walkthrough as appropriate.
- Capture buyer questions about data sources, validation, model caveats, exports, support, procurement, and records retention.
- Name one buyer owner, one decision reviewer, and one Nat Ford support contact before any pilot scope.
- Classify data sensitivity before receiving buyer-provided materials.
- Fill the pilot SOW template and managed-hosting schedule only after the first workflow is agreed.
- Decide whether procurement needs a fresh paid checkout canary before signature.
- Log the outcome: no fit, keep warm, walkthrough scheduled, pilot scope requested, or hold.

## Stop / Hold Conditions

Pause outreach, demo access, or pilot scoping if any of these occur:

- The buyer or recipient reads the demo as NCTC endorsement, approval, partnership, or procurement momentum.
- The buyer needs validated behavioral forecasting, legal/compliance sign-off, certified grant scoring, or autonomous AI planning as the first claim.
- The buyer asks for broad self-serve operation without managed onboarding and support.
- The buyer asks to upload confidential or sensitive data before data terms, access posture, and retention duties are clear.
- The buyer requires SSO, private cloud, agency-cloud deployment, enhanced SLA, or custom security review before a small pilot can be scoped.
- The buyer asks for a public URL, press language, co-marketing, or named-partner claim before written approval.
- The buyer requires fresh same-cycle money-moving billing proof and the current waiver packet is not enough.
- Current proof docs, workspace isolation smoke, or workflow smoke evidence fail or drift from the sales language.
- Scope expands beyond one first workflow without a revised SOW and service schedule.
